webpack配置_hash

  • plugin
  • webpack

本篇对比webpack中常见到的hash种类，主要是hash、chunkHash和contentHash。

介绍

在使用extract-text-webpack-plugin将css样式单独打包时，我们有用到这一一行配置。

plugins: [
  new ExtractTextPlugin("styles.[contentHash:8].css"),
]

这里的contentHash是什么呢？

使用hash，一般结合CDN缓存来用，通过hash，构建生成的文件名自动带上对应的MD5值，如果文件内容改变，那么对应文件的哈希值也会改变，其引用的URL地址也会改变，从而触发CDN服务器从源服务器上拉取对应数据，进而本地缓存更新。

在webpack配置中，hash分为以下三类：

  1. hash
  2. chunkHash
  3. contentHash

第一种跟整个项目的构建相关，只要项目里有文件更改，整个项目构建的hash值都会更改，并且全部文件都共用相同的hash值；

这种方法每一次构建后生成的哈希值都不一样，即使文件内容压根没有改变。这样子是没办法实现缓存效果。

第二种根据不同的入口文件(Entry)进行依赖文件解析、构建对应的chunk，生成对应的哈希值。比较常用的情况是在生产环境里把一些公共库和程序入口文件区分开，单独打包构建，接着我们采用chunkhash的方式生成哈希值，那么只要我们不改动公共库的代码，就可以保证其哈希值不会受影响。

具体来说，比如项目主入口文件index.js及其对应的依赖文件比如index.css由于被打包在同一个模块，所以共用相同的chunkhash值，但是公共库由于是不同的模块，所以有单独的chunkhash。这样子就保证了在线上构建的时候只要文件内容没有更改就不会重复构建。

但也存在问题，如果index.js更改了代码，css文件就算内容没有任何改变，由于是该模块发生了改变，导致css文件也会重复构建。

第三种也就是代码里使用extract-text-webpack-plugin里的contentHash，保证了即使css文件所处的模块里就算其他文件内容改变，只要css文件内容不变，那么就不会重复构建。

解释

为什么会出现这三种hash呢？其实也是因为要解决前端开发和部署优化问题而给出的不同解决方案。这里可以查看fouber大神在《大公司里怎样开发和部署前端代码？》中的回答：

第一步，从原始的前端开发讲起, index.html页面和它的样式文件a.css，用文本编辑器写代码，无需编译，本地预览，确认OK，丢到服务器，等待用户访问。这样，每次用户访问页面都要加载，是不是很影响性能，很浪费带宽。

第二步优化，利用304，让浏览器使用本地缓存。但304是协商缓存，还是要和服务器通信一次(HTTP 304: Not Modified,客户端有缓冲的文档并发出了一个条件性的请求（一般是提供If-Modified-Since头表示客户只想比指定日期更新的文档,而响应头有Last-Modified，对比日期即可，一致就从缓存里拿）。服务器告诉客户，原来缓冲的文档还可以继续使用)。

第三步，强制浏览器使用本地缓存（cache-control/expires），不要和服务器通信。好了，请求方面的优化已经达到变态级别。那问题来了：你都不让浏览器发资源请求了，这缓存咋更新？（cache-controll如果是no-chache说明强制每次请求直接发送给源服务器，而不经过本地缓存版本的校验；max-age>0时，直接从浏览器缓存中提取；max-age<=0 时 向server发送http请求确认,该资源是否有修改有的话，返回200，无的话 返回304）

第四步，通过更新页面中引用的资源路径，让浏览器主动放弃缓存，加载新资源，即href=”a.css?v=1.0.0”，下次上新把版本号修改一下，就更新资源了。但是页面如果引用了3个css，而某次上线只改了其中的a.css，如果所有链接都更新版本，就会导致b.css，c.css的缓存也失效，那岂不是又有浪费了。。。

第五步，让url的修改与文件内容关联，也就是说，只有文件内容变化，才会导致相应url的变更，从而实现文件级别的精确缓存控制。也就是使用hash。

第六步，为了进一步提升网站性能，会把静态资源和动态网页分集群部署，静态资源会被部署到CDN节点上，网页中引用的资源也会变成对应的部署路径。这种情况要采取非覆盖式发布。用文件的摘要信息来对资源文件进行重命名，把摘要信息放到资源文件发布路径中，这样，内容有修改的资源就变成了一个新的文件发布到线上，不会覆盖已有的资源文件。上线过程中，先全量部署静态资源，再灰度部署页面。

总结为：

  1. 配置超长时间的本地缓存［节省带宽，提高性能］
  2. 采用内容摘要作为缓存更新依据［精确的缓存控制］
  3. 静态资源CDN部［优化网络请求］
  4. 资源发布路径实现非覆盖式发布［平滑升级］
